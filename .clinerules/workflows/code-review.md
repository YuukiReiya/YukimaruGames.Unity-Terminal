# コードレビューワークフロー

指定されたファイルまたはクラスをDDD方針・コーディング規約に沿ってレビューする。

---

## Step 1: レビュー対象の確認

ユーザーにレビュー対象を確認する:

- ファイルパス or クラス名
- レビューの観点（全般 / アーキテクチャ重点 / 命名規則重点）

指定がない場合は「全般」として進める。

---

## Step 2: 対象ファイルの読み込み

指定されたファイルを読み込み、以下の情報を把握する:

- クラスの種類（MonoBehaviour / 純粋C# / インターフェース）
- 配置されているフォルダ（レイヤーの確認）
- 依存しているクラス・名前空間

---

## Step 3: アーキテクチャチェック

以下の観点でチェックし、問題点を列挙する:

**レイヤー違反**
- [ ] Domain層に `using UnityEngine` が含まれていないか
- [ ] Domain層に `MonoBehaviour` の継承がないか
- [ ] Presentation層からInfrastructure層を直接参照していないか
- [ ] 依存方向が `Presentation → Application → Domain ← Infrastructure` になっているか

**禁止パターン**
- [ ] `FindObjectOfType()` / `GameObject.Find()` を使っていないか
- [ ] `public` フィールドを直接公開していないか
- [ ] `#region` を使っていないか

---

## Step 4: 命名規則チェック

以下の観点でチェックし、問題点を列挙する:

- [ ] クラス名がPascalCaseか
- [ ] privateフィールドが `_camelCase` か
- [ ] メソッド名がPascalCaseか
- [ ] インターフェース名が `I` + PascalCaseか
- [ ] イベント名が `On` + PascalCaseか

---

## Step 5: コード品質チェック

- [ ] マジックナンバーがハードコードされていないか
- [ ] `Debug.Log` が `#if UNITY_EDITOR` で囲まれているか
- [ ] `// TODO` が残っていないか
- [ ] XMLドキュメントコメントがpublicメソッドに付いているか
- [ ] nullチェックに `?.` 演算子をUnityオブジェクトに使っていないか

---

## Step 6: レビュー結果の報告

以下の形式で結果を報告する:

```
【レビュー結果】対象: ファイル名

■ 問題あり
- [重要度: 高/中/低] 問題の内容と修正案

■ 問題なし
- 該当項目を列挙

■ 総評
全体的なコメント
```

修正が必要な場合、ユーザーの承認を得てから修正を実施する。
