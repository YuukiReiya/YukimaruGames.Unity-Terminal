# コミットワークフロー

変更内容を分析して適切なコミットメッセージを生成し、コミットを実行する。

---

## Step 1: 引数の解析とフィルタリング

**ユーザーが入力したコマンド（例: `/commit *.cs !*.env`）の引数を解析し、以下のルールに従って自律的に情報収集を行うこと:**

### 引数解析アルゴリズム
1.  **包含（Include）**: `!` が付かない引数（例: `*.cs`, `src/`）は、**対象に含めるファイル/ディレクトリ**として扱う。
2.  **除外（Exclude）**: `!` が付く引数（例: `!*.env`, `!/Temp/`）は、**対象から取り除くファイル/ディレクトリ**として扱う。
3.  **デフォルト**: 引数がない場合は、全ての変更ファイルを対象とする。

### 実行手順
- `git --no-pager status` で変更を確認した後、上記ルールを適用して **`git --no-pager diff [包含条件] -- . [除外条件]`** の形式、あるいは個別にフィルタリングして差分を分析する。
- ユーザーに確認することなく、まずは指定条件に合致するファイルの差分を全て読み取ること。


---

## Step 2: 抽出結果の報告

> **注意: 実際の `git add` 前に、必ず以下の内容でユーザーの最終承認を得ること。**
> **注意: このワークフローはActモードであっても必ずユーザーに確認すること。**

- **対象となったファイル**: 解析ルールに基づき、コミット対象として抽出されたファイル一覧。
- **変更の要約**: 抽出されたファイル群の修正内容。
- **分割の提案**: 複数の関心事が含まれる場合、コミットを分けるべきか確認。

---



---

## Step 3: コミットメッセージの生成・確認

差分を分析して以下の形式でコミットメッセージを提案する:

```
feat:   新機能の追加
fix:    バグ修正
refactor: リファクタリング
test:   テストの追加・修正
docs:   ドキュメントの更新
chore:  設定・環境変更
```

例:
```
feat: PlayerHpにダメージ計算ロジックを追加
```

提案内容を出力し、ユーザーに確認し。
承認を得てから次のステップに進む。

---

## Step 4: コミットの実行

**重要：必ず `08-github-instruction.md` の「アイデンティティ確認フロー」に従うこと。**

承認後、**「Step 1で確定した対象ファイルのみ」** を個別に `git add` し、コミットを実行する。
- `git add` 時、除外指定（`!`）されたファイルが絶対に含まれないよう厳密に操作すること。

承認されたメッセージでコミットを実行する:

1.  `.agents/git-credentials.txt` を読み込み、Agent情報を取得（なければユーザーに確認）。
2.  **ローカルコミットのみ**を実行する（`push` は含めないこと）：

```
git add [対象ファイル]
git commit -m "[種別] メッセージ"
```

実行結果を報告する。

---

## Step 5: ユーザー承認によるプッシュ実行

- ユーザーから明示的な `git push` 実行指示がない限り、自動でプッシュしない
- プッシュが必要な場合は、ユーザーに明示的に確認してから実行する
